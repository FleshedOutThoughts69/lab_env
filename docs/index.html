§ 2 — Problem

The missing bridge between theory and production incident response.

Most reliability training stops at diagrams. You read about OOM kills, inode exhaustion, and TCP RSTs—but you never feel them. This lab puts you in front of a real Ubuntu machine, with a real Go service, a real nginx reverse proxy, and real systemd units. You learn the diagnostic pattern, not just the fix.

"Every layer of the filesystem → permissions → process → service → socket → proxy → response chain has a fault that isolates exactly one failure cause."

§ 5 — Lab at a glance

Five commands that cover every operation.

status, validate, fault apply, reset, and history — the entire surface of the lab. No hidden subcommands.

lab status only reconciler — reads + writes state.json
lab validate observation only — never updates state
lab fault apply inject a fault by ID
lab reset restore to conformant — R1 / R2 / R3
lab history ring buffer of state transitions

  • systemd
  • nginx
  • cgroup v2
  • nftables
  • Ubuntu 22.04
